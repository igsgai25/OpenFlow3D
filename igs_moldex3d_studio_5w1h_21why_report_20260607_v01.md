## 1. Moldex3D Studio 3 層 5W1H 架構剖析

### 💻 第一層：UI 與交互表現層 (Qt / QML GUI Tier) [VERIFIED]
*   **Who (受眾與開發角色)**: 模流分析師、模具工程師；由 C++/Qt 介面工程師研發。
*   **What (技術載體)**: 基於 **Qt (C++)** 的整合開發環境，提供工程專案樹管理、材料選擇精靈與動態參數配置面板。
*   **Where (部署位置)**: 用戶本地 Windows 工作站，佔用輕量級內存空間。
*   **When (交互觸發契機)**: 在模擬專案的初期建模、材料設定，以及後期點選探針、向量切片查看分析結果時。
*   **Why (採用 Qt 原因)**: Qt 具備優異的跨平台特性、原生 C++ 的極限執行效能，且其信號與槽 (Signals & Slots) 機制非常適合處理複雜的前後處理視窗連動。
*   **How (運作機制)**: UI 線程獨立運行，透過 Qt 的事件循環與背景運算線程進行通訊，防止三維大規模渲染或檔案載入時介面凍結。

### 🎨 第二層：3D 渲染與後處理核心層 (OpenGL/VTK Renderer Tier) [VERIFIED]
*   **Who (負責角色)**: 科學計算與圖形學渲染工程師。
*   **What (核心組件)**: 基於客製化 **OpenGL / VTK (Visualization Toolkit)** 封裝的 3D 渲染管線。
*   **Where (系統邊界)**: 直接與 GPU 顯示驅動進行底層硬體通訊，運行於專屬的 3D Viewport 面板。
*   **When (渲染觸發)**: 用戶進行網格縮放、切換壓力/溫度等值面 (Isosurface) 雲圖、或播放熔膠充填動態動畫時。
*   **Why (技術選型)**: VTK 具備對百萬級、千萬級有限元素網格（Tetra, Hex, Pyramid）的高效內存佈局與拓樸檢索能力，能流暢渲染複雜流體剪切線與應力分布。
*   **How (資料管線)**: 將有限體積法生成的 `.lch` 二進位數值解，解析為 VTK 的 Unstructured Grid 結構，透過 Shader 進行高效 GPU 渲染。

### ⚙️ 第三層：專利網格與離散求解層 (Mesh Engine & Solver Tier) [VERIFIED]
*   **Who (負責角色)**: 計算幾何學家、數值求解器專家。
*   **What (核心引擎)**: 包含高品質 BLM 網格劃分器與自動非匹配網格（Non-matching Mesh）連接引擎。
*   **Where (計算位置)**: 本地 CPU 多核並行運算（MPI），或提交至 Linux HPC 叢集。
*   **When (觸發時機)**: 用戶完成 CAD 匯入後點選「生成網格」，以及點選「開始分析」進行數值求解時。
*   **Why (工程價值)**: 射出成型壁面固化層厚度僅數微米，普通網格會使傳熱計算完全失真。BLM 網格能精確剖分壁面邊界層，而非匹配網格連接技術能讓流道與產品網格在交界處無需共節點，大幅縮短前處理拼接時間。
*   **How (離散求解)**: 採用 FVM/FEM 混合法離散三維控制方程，透過並行矩陣求解器（如 AMG 代數多重網格）加速大型線性方程組求解。

## 2. Moldex3D Studio 21-Why 底層技術問答 (The 21-Why Chain)

### Q1: 為什麼 Moldex3D Studio 選擇 C++ 作為主力開發語言，而不是 Java 或 Python？
*   **答**: 因為 CAE 前處理網格剖分與後處理 3D 渲染涉及海量幾何數據的內存指針操作，Java 垃圾回收機制會帶來不可控的延遲，而 Python 執行速度過慢，唯有 C++ 能提供極致的硬體控制力與運算效能。

### Q2: 為什麼 GUI 介面不採用 Electron 等 Web 技術，而堅持使用 Qt？
*   **答**: 射出成型網格數量通常達數百萬至上千萬，Electron (Chromium) 的內存開銷極大，且無法與底層 C++ 網格引擎及 VTK 二進位數據指針進行零拷貝 (Zero-copy) 內存級的高速資料交換。

### Q3: 為什麼 Studio 的 3D 視角中必須提供「向量切片 (Vector Cutting)」功能？
*   **答**: 熔膠在流道內的流速與溫度分布在徑向（厚度方向）呈劇烈拋物線分布，外觀雲圖無法展示內部剪切率特徵，必須藉由向量切片即時求解內部流動通量。

### Q4: 為什麼要將求解器 (Solver) 設計為與 Studio GUI 獨立的進程 (Process)？
*   **答**: 求解器運算長達數小時且佔用 100% CPU/GPU 資源。進程隔離能確保即使求解器崩潰或強行終止，Studio 介面依然能安全運行，並便於本地與遠端 HPC 運算節點的動態對接。

### Q5: 為什麼前處理中需要自動判定「分型方向 (Parting Direction)」？
*   **答**: 模具開模方向決定了側向滑塊（Slider）的佈置與澆口設計。自動判定分型線能防止倒扣（Undercut）發生，保證產品能順利脫模。

### Q6: 為什麼冷卻水道模擬需要引進「干涉預覽」？
*   **答**: 水道距離模穴太近會造成模具局部強度不足甚至破裂，太遠則冷卻失效。干涉預覽能在設計階段自動預警水道與模具結構件的碰撞，防止後段加工返工。

### Q7: 為什麼自動網格連接需要「非匹配網格 (Non-matching Mesh)」技術？
*   **答**: 傳統 CAE 要求流道與產品網格在接觸面上「節點一對一匹配」，這在幾何複雜時需要耗費幾天人工去對齊網格。非匹配技術利用交介面上的數值通量插值（Mortar Method），允許流道與產品網格自由拼接，節省 90% 前處理時間。

### Q8: 為什麼材料資料庫（Material Database）格式使用 SQLite？
*   **答**: 塑料材料數據庫包含 9,000+ 筆記錄，每筆記錄包含複雜的 Cross-WLF 與 Tait PVT 多項式係數。SQLite 是輕量、免配置且支援 SQL 高速查詢的嵌入式關係型資料庫，非常適合桌面端嵌入。

### Q9: 為什麼 Studio 必須支持「多點探針一次到位」？
*   **答**: 在多型腔射出中，工程師需要比對多個型腔在同一時間點的壓力降與熔膠前沿溫度。多點探針能即時輸出多通道歷史曲線，便於定量評估流動平衡度。

### Q10: 為什麼三維網格生成中要使用「五角錐 (Pyramid) 網格」過渡？
*   **答**: 壁面 BLM 網格通常是六面體（Prism）或四角柱，而型腔內部網格常為四面體（Tetra）。五角錐網格能作為界面過渡，無縫連接結構網格與非結構網格，防止離散計算發散。

### Q11: 為什麼求解器在充填初期容易發生數值發散？
*   **答**: 充填初期熔膠剛射出，流速極高且前沿與空氣介面存在強烈的相變不連續性，流體控制方程在該處具有極強的奇異性，需要精細調整時間步長（Time Step）與鬆弛因子。

### Q12: 為什麼必須模擬「固化層 (Frozen Layer)」的動態增長？
*   **答**: 熔膠接觸冷模壁會瞬間冷卻結凍，使流動通道變窄，增加射出阻力。如果固化層厚度預測失真，計算出的射出壓力與鎖模力將會偏低 20-30%。

### Q13: 為什麼發泡成型（Foaming）的模擬需要旋轉設定介面？
*   **答**: 物理發泡在模穴內的氣泡成核與生長受到模具開模移動（如 Breathing Mold 核心回退）與重力方向的強烈影響，旋轉設定能讓重力與運動向量精確對齊空間座標。

### Q14: 為什麼 RTM 模擬要支援導入包含 Ply 資訊的 INP 檔案？
*   **答**: 樹脂轉注成型 (RTM) 中，碳纖維織物的層數（Plys）與鋪角（Fiber Angle）決定了樹脂在織物內的滲透率 (Permeability) 各向異性，INP 檔能精確載入複合材料的結構資訊。

### Q15: 為什麼 Studio 的 Max/Min 標記需要支持「方向鎖定」？
*   **答**: 當工程師旋轉三維模型時，如果標記數值與指引線隨著視角隨機飄移，會遮擋關鍵流動特徵。方向鎖定能確保空間可視化標註的清晰度。

### Q16: 為什麼 3D 模擬中的壓降計算常常比實際現場偏大？
*   **答**: 因為模擬通常假設模壁是絕對剛體且流阻不變，而實際射出高壓下模具會發生微小的「彈性開模」，局部間隙變大使得實際壓降略小於模擬值。

### Q17: 為什麼 Moldex3D 材料精靈需要支持粉末材料建置？
*   **答**: 金屬粉末射出成型 (MIM) 與陶瓷粉末射出 (CIM) 中，粉末與黏結劑存在「兩相分離 (Powder-Binder Segregation)」現象，必須建置專屬的粉末材料庫才能模擬相分離引起的局部強度不均。

### Q18: 為什麼 Studio 的後處理渲染需要支持 GPU 著色器 (Shader)？
*   **答**: 千萬級網格的流速向量箭頭如果用 CPU 逐個計算位置與渲染，畫面幀率會跌至 <5 FPS。使用 GPU 頂點著色器 (Vertex Shader) 可以直接在顯卡中並行計算向量箭頭的位置與指向，流暢維持 60 FPS 以上的交互。

### Q19: 為什麼 2026 版本需要一鍵串聯 Molding Window 與 iMolding Advisor？
*   **答**: molding window 提供理論靜態成型窗口，而 iMolding advisor 結合真實機台動態響應。兩者串聯能實現「先理論尋優，後實機修正」的雙步校準，大幅縮短現場調機工時。

### Q20: 為什麼網格自動修補（Mesh Repairing）是前處理的核心痛點？
*   **答**: CAD 導入時常因公差問題產生破面、自由邊（Free Edges）或網格自交。若不自動修復這些幾何拓樸錯誤，三維 Solid 網格生成引擎將無法成功封閉並實體化。

### Q21: 為什麼打通 Moldiverse 雲端生態系是 Studio 的未來戰略？
*   **答**: CAE 用戶不僅需要單機工具，更需要即時的材料參數檢索、自動化優化 App、以及全球學術論壇的支援。Moldiverse 雲端生態系將 Studio 桌面端轉化為連網的智慧終端，極大化提升了用戶黏著度與生態壁壘 [VERIFIED]。
